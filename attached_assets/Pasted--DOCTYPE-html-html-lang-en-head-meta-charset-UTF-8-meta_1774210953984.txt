    <div class="st">What "good suggestions" actually means</div>
    <div class="ss">This shapes every decision in the spec below. Keep it visible.</div>

    <div class="compare">
      <div class="compare-card">
        <div class="compare-head old">❌ What the current system does</div>
        <div class="compare-body">
          Finds the <strong>mathematically balanced group</strong> based on team average scores right now, with no awareness of what happened earlier in the session.<br><br>
          Result: the same 4 players who are always near the front of the queue get the best games, while players who joined slightly later get fewer games and worse matchups. Current data shows a <strong>5.8-game spread</strong> between the most and least-played player per session.
        </div>
      </div>
      <div class="compare-card">
        <div class="compare-head new2">✓ What the new system must do</div>
        <div class="compare-body">
          Finds the <strong>best competitive game for this moment</strong>, while actively tracking who has played the least this session and surfacing them first.<br><br>
          Result: by the end of a session, every player has had roughly the same number of games (target: <strong>max 2-game spread</strong>), and every game was genuinely competitive. The admin's job is to confirm, not to think.
        </div>
      </div>
    </div>

    <div class="hl g"><strong>Admin's role:</strong> The admin sees ordered suggestions, best first. They confirm the top option in most cases. They override when they know something the algorithm doesn't — a player is tired, two players are arguing, a new player needs an easier game. The system advises. The admin decides.</div>
  </div>

  <div class="section">
    <div class="sl">01</div>
    <div class="st">Session State — What to Track Per Player</div>
    <div class="ss">The algorithm needs these values in memory at all times. Rebuild from game history on server restart. Reset to zero when a new session starts.</div>

    <div class="tw">
      <table>
        <thead><tr><th>Field</th><th>Type</th><th>What it tracks</th><th>Updated when</th></tr></thead>
        <tbody>
          <tr><td class="hl2">games_this_session</td><td class="code">integer</td><td>Total games played since this session started</td><td>After every game completes</td></tr>
          <tr><td class="hl2">consecutive_games</td><td class="code">integer</td><td>Games played in a row without a break</td><td>After every game. Halved after 1 round out. Zeroed after 2 rounds out.</td></tr>
          <tr><td class="hl2">rounds_waited</td><td class="code">integer</td><td>How many full court rounds have completed since this player last played</td><td>Incremented every time a court round completes and this player was not in it</td></tr>
          <tr><td class="hl2">is_sitting_out</td><td class="code">boolean</td><td>Voluntary sit-out flag</td><td>Toggled by admin. Auto-cleared after 1 round.</td></tr>
          <tr><td class="hl2">last_partners</td><td class="code">player_id[]</td><td>IDs of partners from last 2 games</td><td>After every game</td></tr>
          <tr><td class="hl2">last_opponents</td><td class="code">player_id[]</td><td>IDs of opponents from last 2 games</td><td>After every game</td></tr>
          <tr><td class="hl2">session_game_quality</td><td class="code">float</td><td>Average skill gap of all games played this session. Lower = better games.</td><td>After every game</td></tr>
        </tbody>
      </table>
    </div>

    <div class="hl a"><strong>Session average games:</strong> Calculate dynamically as <code style="font-family:'Space Mono',monospace;font-size:11px;">total_games_played_this_session / active_players_count</code>. This is the fairness target. Any player below this average is under-served and should be prioritised.</div>

  <!-- ═══ 02 PLAYER PRIORITY SCORE ═══ -->
  <div class="section">
    <div class="sl">02</div>
    <div class="st">Player Priority Score — Revised Formula <span class="badge bchg" style="margin-left:8px;vertical-align:middle;">Changed</span></div>
    <div class="ss">The most important change from the previous spec. Queue position weight is reduced significantly. Session equity is added as the dominant fairness driver.</div>

    <div class="pbox">
      <div class="peq">
<span class="pk">priority</span>  =  (queue_position  <span class="pk">×  25</span>)              <span class="pc"> ← reduced from ×100. Still matters but no longer dominates.</span>
           +  (consecutive_games  <span class="pk">×  10</span>)          <span class="pc"> ← only if consecutive_games ≥ 2 (needsRest)</span>
           +  (total_games_ever  <span class="pk">×  0.1</span>)          <span class="pc"> ← tiny tiebreaker only</span>
           <span class="ps">−  (rounds_waited  ×  18)</span>                <span class="pc"> ← rewards patience</span>
           <span class="pnew">−  (session_games_deficit  ×  40)</span>        <span class="pc"> ← NEW dominant fairness driver</span>

<span class="pc">session_games_deficit  =  session_avg_games  −  player.games_this_session
If deficit is negative (player played MORE than average), this term adds to priority (penalises them).
If deficit is positive (player played LESS than average), this term subtracts (boosts them up).

Lower priority score  =  Higher priority  =  Suggested first</span>
      </div>
    </div>

    <div class="compare" style="margin-top:16px;">
      <div class="compare-card">
        <div class="compare-head old">Old formula problem</div>
        <div class="compare-body">Queue position weight of ×100 meant a player at position 25 needed to wait <strong>139 rounds</strong> to overcome the penalty vs position 1. The waiting bonus (×18) was too weak to matter. Players who joined late were systematically under-served for the entire session.</div>
      </div>
      <div class="compare-card">
        <div class="compare-head new2">New formula result</div>
        <div class="compare-body">Session games deficit weight of ×40 means a player with 2 fewer games than average gets a <strong>−80 point boost</strong> — strong enough to pull them ahead of a queue-position 3 penalty (+75). Players who've played less are actively surfaced. Target: ≤2 game spread per session.</div>
      </div>
    </div>

    <div class="cl">Priority score examples — 25 player session, avg 4 games played</div>
    <div class="cb"><pre><span class="cm">// Player A: position 3, played 2 games (2 below avg), 1 round waited, 0 consecutive</span>
priority_A = (3 × 25) + 0 − (1 × 18) − (2 × 40) = 75 + 0 − 18 − 80 = <span class="pnew">−23</span>  ← very high priority

<span class="cm">// Player B: position 1, played 6 games (2 above avg), 0 waited, 2 consecutive</span>
priority_B = (1 × 25) + (2 × 10) − 0 + (2 × 40) = 25 + 20 + 80 = <span class="ps">125</span>  ← low priority despite being first

<span class="cm">// Player C: position 8, played 4 games (at avg), 0 waited, 0 consecutive</span>
priority_C = (8 × 25) + 0 − 0 − 0 = <span class="nm">200</span>  ← mid priority</pre></div>
  </div>

  <!-- ═══ 03 LONE OUTLIER DETECTION ═══ -->
  <div class="section">
    <div class="sl">03</div>
    <div class="st">Lone Outlier Detection <span class="badge bnew" style="margin-left:8px;vertical-align:middle;">New</span></div>
    <div class="ss">Before generating any suggestions, scan the active player pool for tiers with only one player. These players cannot be matched within their tier and must be handled explicitly.</div>

    <div class="steps">
      <div class="step">
        <div class="snc"><span class="sn">1</span></div>
        <div class="sb">
          <div class="stitle">Count active players per tier</div>
          <div class="sdesc">At suggestion generation time, count how many players in the queue belong to each tier. A tier with <strong>fewer than 4 players</strong> cannot fill a court on its own.</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">2</span></div>
        <div class="sb">
          <div class="stitle">Flag lone outliers (1 player in their tier)</div>
          <div class="sdesc">Any player who is the <strong>only player</strong> in their tier this session is a lone outlier. They are <strong>excluded from all auto-generated suggestions</strong>. Show a dedicated notice card at the top of the suggestions panel — not buried, not subtle. Example: <em>"Sriram (Advanced, 130) is the only Advanced player this session. Assign him manually by dragging to a court, or ask him to wait."</em></div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">3</span></div>
        <div class="sb">
          <div class="stitle">Flag small tiers (2–3 players)</div>
          <div class="sdesc">If a tier has 2 or 3 players, they cannot fill a full court. Show a softer notice: <em>"Only 3 Lower-Int players available — they will be included in overflow suggestions."</em> These players are still eligible to appear in mixed-tier suggestions (Section 05, overflow rules) but are not the primary suggestion.</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">4</span></div>
        <div class="sb">
          <div class="stitle">Admin can manually override</div>
          <div class="sdesc">The admin can drag any lone outlier player into a court's team assignment panel at any time. When this happens, the system should show the resulting within-team spread and tier gap as a warning but not prevent the assignment. <strong>The admin has final authority.</strong></div>
        </div>
      </div>
    </div>

    <div class="hl r"><strong>Today's session example:</strong> Sriram (Advanced, 130) is the only Advanced player. All 5 suggestions currently shown pair him with Intermediate players, producing within-team spreads of 50–53 points. Under this rule, <strong>none of those 5 suggestions would be generated</strong>. Instead, a notice card explains Sriram's situation and the suggestions show 3 excellent Intermediate-only games.</div>
  </div>

  <!-- ═══ 04 WITHIN-TEAM SPREAD FILTER ═══ -->
  <div class="section">
    <div class="sl">04</div>
    <div class="st">Within-Team Spread Hard Filter <span class="badge bchg" style="margin-left:8px;vertical-align:middle;">Critical</span></div>
    <div class="ss">Applied before any suggestion is shown. A split is invalid if either team's players are too far apart in score. This prevents "average masking" — where a strong player carrying a weak one creates a misleadingly balanced team average.</div>

    <div class="tw">
      <table>
        <thead><tr><th>Within-team spread</th><th>Decision</th><th>Reason</th></tr></thead>
        <tbody>
          <tr><td class="good">0 – 10 pts</td><td class="good">✓ Excellent</td><td>Genuine partnership — both players contribute equally</td></tr>
          <tr><td class="warn">11 – 20 pts</td><td class="warn">✓ Acceptable</td><td>One player slightly stronger but still a real team</td></tr>
          <tr><td class="bad">21 – 30 pts</td><td class="bad">❌ Reject</td><td>One player will carry — poor experience for both</td></tr>
          <tr><td class="bad">&gt; 30 pts</td><td class="bad">❌ Reject</td><td>Not a real partnership. Effectively 1v2 on key rallies.</td></tr>
        </tbody>
      </table>
    </div>

    <div class="cl">Implementation</div>
    <div class="cb"><pre><span class="kw">const</span> MAX_WITHIN_TEAM_SPREAD = <span class="nm">20</span>;

<span class="kw">function</span> <span class="fn">isValidSplit</span>(team1, team2) {
  <span class="kw">const</span> spread1 = Math.max(...team1.map(p => p.score)) - Math.min(...team1.map(p => p.score));
  <span class="kw">const</span> spread2 = Math.max(...team2.map(p => p.score)) - Math.min(...team2.map(p => p.score));
  <span class="kw">return</span> spread1 <= MAX_WITHIN_TEAM_SPREAD && spread2 <= MAX_WITHIN_TEAM_SPREAD;
}

<span class="cm">// If all 3 possible splits for a group fail this filter,</span>
<span class="cm">// the group itself is invalid. Try the next priority combination.</span>
<span class="cm">// If no group can form a valid split, widen the candidate window by 4 and retry once.</span>
<span class="cm">// If still no valid split, relax to 25pts and show a "Compromised" warning label.</span></pre></div>

    <div class="hl">Each suggestion card must display the within-team spread for both teams. Colour code it: <strong style="color:var(--success);">green ≤10</strong>, <strong style="color:var(--amber);">amber 11–20</strong>, <strong style="color:var(--red);">red &gt;20</strong> (should never appear after this filter is live, but useful during transition).</div>
  </div>

  <!-- ═══ 05 SUGGESTION GENERATION ═══ -->
  <div class="section">
    <div class="sl">05</div>
    <div class="st">Suggestion Generation — Full Logic</div>
    <div class="ss">How the engine produces the ordered list of suggestions shown to the admin for a single court.</div>

    <div class="steps">
      <div class="step">
        <div class="snc"><span class="sn">1</span></div>
        <div class="sb">
          <div class="stitle">Run lone outlier detection (Section 03)</div>
          <div class="sdesc">Remove lone outliers from the eligible pool before doing anything else. Show their notice cards at the top of the UI.</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">2</span></div>
        <div class="sb">
          <div class="stitle">Build the candidate window</div>
          <div class="sdesc">From the active queue (excluding sitting-out players and those already on a court), take the front <strong>floor(queue_length × 50%)</strong>, clamped to a minimum of 8 and maximum of 24 players. This is the candidate pool.</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">3</span></div>
        <div class="sb">
          <div class="stitle">Calculate priority score for each candidate (Section 02)</div>
          <div class="sdesc">Apply the revised formula including the session_games_deficit term. Sort candidates by priority score (lowest first).</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">4</span></div>
        <div class="sb">
          <div class="stitle">Identify the anchor tier</div>
          <div class="sdesc">The player with the <strong>lowest priority score</strong> (highest priority) in the candidate pool defines the target tier. The engine builds suggestions primarily from their tier first.</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">5</span></div>
        <div class="sb">
          <div class="stitle">Generate up to 20 four-player combinations from the tier pool</div>
          <div class="sdesc">Filter candidates to the anchor tier. Generate all unique combinations of 4. If fewer than 4 in the tier, apply overflow (Section 06). For each combination of 4, generate all 3 possible team splits and apply the within-team spread filter (Section 04). Discard any combination where no valid split exists.</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">6</span></div>
        <div class="sb">
          <div class="stitle">Score and rank all valid combinations</div>
          <div class="sdesc">For each valid combination, take its best valid split and score it using the 5-factor ranking (Section 07). Sort all combinations by their score. Return the top <strong>5 suggestions</strong> to the admin — not 15. Five is enough; more creates decision paralysis.</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">7</span></div>
        <div class="sb">
          <div class="stitle">Label the top suggestion clearly</div>
          <div class="sdesc">The first suggestion gets a <strong>"Best Match"</strong> label (green). Suggestions 2–5 get an <strong>"Option"</strong> label. The admin should be able to tap "Assign to Court" on Option 1 in most sessions without needing to read further.</div>
        </div>
      </div>
    </div>

  <!-- ═══ 07 SUGGESTION RANKING ═══ -->
  <div class="section">
    <div class="sl">07</div>
    <div class="st">Suggestion Ranking — 5-Factor Sort <span class="badge bnew" style="margin-left:8px;vertical-align:middle;">Updated</span></div>
    <div class="ss">After filtering, rank valid splits using this five-level sort. Each level only applies if the previous level is a tie (difference &lt; 0.5).</div>

    <div class="tw">
      <table>
        <thead><tr><th>Priority</th><th>Factor</th><th>What it measures</th><th>Direction</th></tr></thead>
        <tbody>
          <tr>
            <td class="hl2" style="color:var(--blue);">1st</td>
            <td class="hl2">Tier dispersion</td>
            <td>Max tier index minus min tier index across all 4 players. 0 = all same tier.</td>
            <td>Lower = better</td>
          </tr>
          <tr>
            <td class="hl2" style="color:var(--green);">2nd</td>
            <td class="hl2">Skill gap</td>
            <td>Absolute difference between Team 1 and Team 2 average scores. Primary balance metric.</td>
            <td>Lower = better</td>
          </tr>
          <tr>
            <td class="hl2" style="color:var(--amber);">3rd</td>
            <td class="hl2">Session equity score</td>
            <td>Sum of (session_games_deficit) for all 4 players. Groups with more under-served players are preferred.</td>
            <td>Lower = better (more deficit = more deserving)</td>
          </tr>
          <tr>
            <td class="hl2" style="color:var(--red);">4th</td>
            <td class="hl2">Partner rotation penalty</td>
            <td>+15 per repeated pair this session (cap +30/pair, +60 total). Prefer fresh pairings.</td>
            <td>Lower = better</td>
          </tr>
          <tr>
            <td class="hl2" style="color:var(--dim);">5th</td>
            <td class="hl2">Score variance</td>
            <td>Variance of all 4 scores around the group mean. Tighter groups play more evenly.</td>
            <td>Lower = better</td>
          </tr>
        </tbody>
      </table>
    </div>

    <div class="cl">Ranking function</div>
    <div class="cb"><pre><span class="kw">function</span> <span class="fn">scoreSplit</span>(split, sessionState, partnerHistory) {
  <span class="kw">const</span> { team1, team2 } = split;
  <span class="kw">const</span> all4 = [...team1, ...team2];

  <span class="cm">// 1. Tier dispersion</span>
  <span class="kw">const</span> tierIndices = all4.map(p => TIER_INDEX[p.current_tier]);
  <span class="kw">const</span> tierDisp    = Math.max(...tierIndices) - Math.min(...tierIndices);

  <span class="cm">// 2. Skill gap between team averages</span>
  <span class="kw">const</span> avg1     = team1.reduce((s,p) => s+p.score, 0) / 2;
  <span class="kw">const</span> avg2     = team2.reduce((s,p) => s+p.score, 0) / 2;
  <span class="kw">const</span> skillGap = Math.abs(avg1 - avg2);

  <span class="cm">// 3. Session equity — prefer groups with more under-served players</span>
  <span class="kw">const</span> sessionAvg    = sessionState.avgGamesPlayed;
  <span class="kw">const</span> equityScore   = all4.reduce((s,p) => s + (sessionAvg - p.games_this_session), 0);
  <span class="cm">// Higher equityScore = more under-served players = BETTER = lower sort value needed</span>
  <span class="kw">const</span> equityRank    = -equityScore;  <span class="cm">// negate so lower = better in sort</span>

  <span class="cm">// 4. Partner rotation penalty</span>
  <span class="kw">const</span> pairs   = [[team1[0].id, team1[1].id], [team2[0].id, team2[1].id]];
  <span class="kw">let</span>   penalty = 0;
  <span class="kw">for</span> (<span class="kw">const</span> pair <span class="kw">of</span> pairs) {
    penalty += Math.min((partnerHistory[pairKey(pair)] ?? 0) * 15, 30);
  }
  penalty = Math.min(penalty, 60);

  <span class="cm">// 5. Score variance</span>
  <span class="kw">const</span> mean     = all4.reduce((s,p) => s+p.score, 0) / 4;
  <span class="kw">const</span> variance = all4.reduce((s,p) => s+(p.score-mean)**2, 0) / 4;

  <span class="kw">return</span> { split, tierDisp, skillGap, equityRank, penalty, variance };
}

<span class="kw">function</span> <span class="fn">rankSuggestions</span>(validSplits, sessionState, partnerHistory) {
  <span class="kw">return</span> validSplits
    .map(s => <span class="fn">scoreSplit</span>(s, sessionState, partnerHistory))
    .sort((a, b) =>
      a.tierDisp   - b.tierDisp   ||
      a.skillGap   - b.skillGap   ||
      a.equityRank - b.equityRank ||  <span class="cm">// NEW: session equity</span>
      a.penalty    - b.penalty    ||
      a.variance   - b.variance
    )
    .slice(0, 5);  <span class="cm">// return top 5 only</span>
}</pre></div>
  </div>
