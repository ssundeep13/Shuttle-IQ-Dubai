What ShuttleIQ is
ShuttleIQ is a managed badminton session platform based in Dubai. It is not a court booking tool. The model is: ShuttleIQ rents courts at venues, provides shuttles, and employs a host (Shannon) who manages player rotation and matchmaking during live sessions. Players book spots in advance via the marketplace. During sessions, Shannon runs the admin panel to manage courts, queues, and game flow. Players access the platform via the marketplace (player-facing side).

Frontend: React + Vite, TanStack Query for data fetching, shadcn/ui components, Tailwind CSS
Backend: Express + TypeScript
Database: PostgreSQL via Drizzle ORM, hosted on Neon
Payments: Ziina webhooks
Email: Resend
AI matchmaking: Claude Sonnet called server-side via the Anthropic API
Background jobs: a scheduler already running via startScheduler() in server/scheduler.ts
The app serves both an admin panel (Shannon's interface) and a marketplace (player-facing interface) from the same Express server

What we are about to build — the player gameplay flow
We are adding a self-service gameplay experience for players on the marketplace side. The flow works like this:

Player arrives at the venue, opens the app, and checks in
They are shown their next game immediately if a court is available, or a waiting screen showing their upcoming game, court, partners, and opponents if not
Once their game is on court they see a live game screen with court, teammates, opponents, and a timer
When the game ends, any one of the 4 players on that court can submit the score
The score is recorded immediately — no confirmation from other players required
The other 3 players can tap a flag button on the completed result if they believe the score is wrong. This creates a score dispute and alerts Shannon to resolve manually
After submitting or seeing a confirmed score, the player is shown their next game
At the end of the session, after the last game is recorded, the player sees a session summary screen

Key constraints for everything we build

Shannon's existing admin flow must not be broken or changed. We are adding to the marketplace side only
When a game ends and a score is submitted, the AI matchmaking suggestion must fire in the background — it must never block the player's screen or the API response
The matchmaking suggestion should auto-approve after 90 seconds unless Shannon overrides it in her admin panel. This timer lives in the existing scheduler
All new player-facing screens live within the existing marketplace routing structure
TanStack Query polling interval on the active game screen should be set to 5 seconds
No new npm packages should be installed unless absolutely necessary — use what is already in the codebase
All new components must follow the existing shadcn/ui and Tailwind patterns already used in the marketplace
Skill score updates must be wrapped in a Drizzle database transaction — all reads, calculations, and writes for all 4 players in a game must succeed together or roll back together
