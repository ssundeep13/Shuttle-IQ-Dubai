  <div class="section">
    <div class="sl">Part 1 — Foundations</div>
    <div class="st">The One Goal</div>
    <div class="ss">Every decision in this spec flows from a single goal. Keep it visible.</div>
    <div class="hl g"><strong>Goal:</strong> By the end of a session, every player has played roughly the same number of games (target: max 2-game spread), and every game was genuinely competitive. The admin sees ordered suggestions, best first, and confirms. They override when they know something the algorithm doesn't. The system advises. The admin decides. There is no auto-assign.</div>
    <div class="hl r"><strong>Current problem:</strong> Real session data shows a 5.8-game spread between the most and least-played player per session. The previous implementation applied tier filtering at the combination level — after mixing all players together. This allowed wrong-tier players into suggestions whenever their inclusion happened to produce a mathematically similar team average. The rebuild applies every filter at the pool level first, before any combination is generated.</div>
  </div>

  <div class="section">
    <div class="sl">Part 3 — Session State</div>
    <div class="st">What to Track Per Player (In-Memory, No DB)</div>

    <div class="tw">
      <table>
        <thead><tr><th>Field</th><th>Type</th><th>Purpose</th><th>Updated when</th></tr></thead>
        <tbody>
          <tr><td class="b">gamesThisSession</td><td class="code">number</td><td>Games played today. Drives session equity.</td><td>+1 after every game this player completes</td></tr>
          <tr><td class="b">consecutiveGames</td><td class="code">number</td><td>Streak without a break. Triggers rest penalty.</td><td>+1 when played. Halved after 1 round out. Zero after 2.</td></tr>
          <tr><td class="b">gamesWaited</td><td class="code">number</td><td>Games ended without this player since they last played.</td><td>+1 each time any game ends and this player was not in it. <strong>Not incremented during voluntary sit-out.</strong></td></tr>

    <div class="hl a"><strong>gamesWaited weight is ×6, not ×18.</strong> The counter increments once per game (not per round). With 3 courts, one full parallel round produces +3 to gamesWaited. At ×6, that equals an effective −18 per round — matching the intended weight. Using ×18 would triple the effect and create extreme priority distortion.</div>
  </div>

  ══════════════════════════════════════════════════════════ -->

  <div class="section">
    <div class="sl">Part 4 — Priority Formula</div>
    <div class="st">calculatePlayerPriority — Lower Score = Higher Priority</div>

    <div class="pbox">
      <div class="peq">
<span class="pk">priority</span>  =  (queuePosition  <span class="pk">× 25</span>)
           +  (consecutiveGames  <span class="pk">× 10</span>)   <span class="pc">← only when consecutiveGames ≥ 2</span>
           +  (totalGamesEver  <span class="pk">× 0.1</span>)    <span class="pc">← tiny tiebreaker only</span>
           <span class="ps">−  (gamesWaited  × 6)</span>           <span class="pc">← patience reward</span>
           <span class="pnew">−  (sessionDeficit  × 40)</span>        <span class="pc">← NEW dominant fairness driver</span>

<span class="pc">sessionDeficit  =  sessionAvgGames  −  player.gamesThisSession
  Positive deficit (played less than avg)  →  subtracts from priority  →  boosted up
  Negative deficit (played more than avg)  →  adds to priority  →  pushed down
sessionAvgGames  =  totalGamesPlayedThisSession / eligiblePlayerCount  (recalculated each call)</span></pre>
    </div>

    <div class="ex">
      <div class="ex-head neutral">Worked examples — 25 players in session, session avg = 4 games played</div>
      <div class="ex-body">
        <div class="tw" style="margin:0;">
          <table>
            <thead><tr><th>Player situation</th><th>Calculation</th><th>Priority</th><th>Result</th></tr></thead>
            <tbody>
              <tr><td>Pos 3, played 2 (deficit +2), waited 4, consec 0</td><td>(3×25)+(0) −(4×6) −(2×40)</td><td class="good">−29</td><td class="good">Highest — surfaced first</td></tr>
              <tr><td>Pos 1, played 6 (deficit −2), waited 0, consec 3</td><td>(1×25)+(3×10) −0 −(−2×40)</td><td class="bad">135</td><td class="bad">Pushed down despite pos 1</td></tr>
              <tr><td>Pos 8, at average, waited 0, consec 0</td><td>(8×25)+0−0−0</td><td class="warn">200</td><td>Mid priority</td></tr>
              <tr><td>Pos 12, played 1 less than avg, waited 2, consec 0</td><td>(12×25)+0−(2×6)−(1×40)</td><td class="warn">248</td><td>Mid-low priority</td></tr>
            </tbody>
          </table>
        </div>
      </div>
    </div>

    <div class="hl"><strong>Why queuePosition weight dropped from ×100 to ×25:</strong> At ×100, a player at position 25 needed 139 rounds of waiting to overcome the penalty vs position 1. The algorithm was effectively ignoring all fairness logic. At ×25, session equity (×40) can meaningfully override queue position, which is the entire point.</div>
  </div>

  <div class="section">
    <div class="sl">Part 5 — The Exact Sequence</div>
    <div class="st">generateSuggestions — Do These Steps in This Exact Order</div>
    <div class="ss">This is the section that was implemented incorrectly before. The previous code mixed players from different tiers before filtering. Every step below must happen in the stated order. Steps 1–3 are pool-level operations. Steps 4–7 are combination-level operations. Pool operations come first, always.</div>

    <div class="steps">
      <div class="step">
        <div class="snc"><span class="sn">1</span></div>
        <div class="sb">
          <div class="stitle">Build the eligible pool</div>
          <div class="sdesc">Start from the full active queue. Remove: players with <strong>isSittingOut = true</strong>, players already assigned to an active game. This is your eligible pool. Do not touch tier yet.</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">2</span></div>
        <div class="sb">
          <div class="stitle">Calculate priority score for every player in the eligible pool</div>
          <div class="sdesc">Apply the formula from Part 4 to every player. Compute <strong>sessionAvgGames</strong> fresh at this moment as totalGamesThisSession / eligiblePool.length. Sort the pool by priority score ascending (lowest first = highest priority).</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">3</span></div>
        <div class="sb">
          <div class="stitle">POOL-LEVEL: Count players per tier. Detect lone outliers. Set overflow flags.</div>
          <div class="sdesc">
            Count how many players in the eligible pool belong to each tier.<br><br>
            <strong>Lone outlier:</strong> any tier with exactly 1 player. Exclude that player from all normal suggestion generation. They get separate Stretch Match handling (Part 7).<br><br>
            <strong>Overflow flag per tier:</strong> set <code style="font-family:'Space Mono',monospace;font-size:11px;">needsOverflow[tier] = true</code> when that tier has fewer than 4 eligible players (after removing lone outliers). When <code style="font-family:'Space Mono',monospace;font-size:11px;">needsOverflow[tier] = false</code>, that tier has enough players to fill a court entirely on its own — no mixing allowed.
          </div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">4</span></div>
        <div class="sb">
          <div class="stitle">POOL-LEVEL: Build the candidate window for the target tier</div>
          <div class="sdesc">
            The <strong>anchor player</strong> is the first player in the priority-sorted pool (excluding lone outliers). Their tier is the <strong>target tier</strong>.<br><br>
            Build the candidate window: take the top <strong>floor(eligiblePool.length × 50%)</strong> players from the priority-sorted list, clamped min 8 / max 24.<br><br>
            <strong>If needsOverflow[targetTier] = false:</strong> filter the candidate window to keep only players whose tier === targetTier. This is the final candidate pool. No other tiers enter.<br><br>
            <strong>If needsOverflow[targetTier] = true:</strong> filter to targetTier players, then add up to 2 players from ADJACENT_TIER[targetTier] (highest priority first from that adjacent tier). No other tiers enter.
          </div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">5</span></div>
        <div class="sb">
          <div class="stitle">COMBINATION-LEVEL: Generate groups of 4 and filter splits</div>
          <div class="sdesc">
            Generate all unique combinations of 4 from the candidate pool (up to 20 combinations, prioritise those containing the highest-priority players).<br><br>
            For each group of 4, generate all 3 possible team splits. Apply <strong>isValidSplit</strong> (Part 6) to each. If all 3 splits fail, discard this group. If no group passes after all combinations are exhausted, retry once with spread limit relaxed to 25 and mark results <code style="font-family:'Space Mono',monospace;font-size:11px;">isCompromised: true</code>.
          </div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">6</span></div>
        <div class="sb">
          <div class="stitle">COMBINATION-LEVEL: Rank all valid splits using 5-factor sort</div>
          <div class="sdesc">For each valid group, take its best valid split. Score it using the ranking in Part 8. Sort all groups by their best split score. Return the top <strong>5 suggestions</strong>.</div>
        </div>
      </div>
      <div class="step">
        <div class="snc"><span class="sn">7</span></div>
        <div class="sb">
          <div class="stitle">Apply labels and return</div>
          <div class="sdesc">Label the first suggestion "Best Match" (green) if gap ≤ 8 AND tierDispersion = 0. Otherwise label it "Closest Available" (amber). If the top label is "Closest Available", label suggestions 2–5 as "Alternative 2" through "Alternative 5". Otherwise label them "Option 2" through "Option 5". Return suggestions + loneOutliers + stretchMatches arrays.</div>
        </div>
      </div>
    </div>

    <div class="hl r"><strong>The previous bug in plain language:</strong> The old code computed combinations from the full mixed pool, then checked tiers inside the combination loop. This meant a group like [Abhiram(74, Lower-Int), Arjun(94, Upper-Int), Alan(109, Upper-Int), Basil(101, Upper-Int)] passed because it had a "majority" of Upper-Int players. The new code never puts Abhiram in the Upper-Int candidate pool in the first place, so this group is never generated.</div>

       PART 6 — SPREAD FILTER

  <div class="section">
    <div class="sl">Part 6 — Within-Team Spread Filter</div>
    <div class="st">isValidSplit — Applied to Every Split Before Ranking</div>
    <div class="ss">Prevents "average masking" — where pairing a very strong player with a very weak one creates a team average that looks balanced but produces an unplayable game. The within-team spread is max(scores) − min(scores) within one team.</div>

    <div class="tw">
      <table>
        <thead><tr><th>Context</th><th>Max spread allowed</th><th>Label if used</th></tr></thead>
        <tbody>
          <tr><td class="b">Normal suggestions</td><td class="good">20 pts</td><td>No label needed</td></tr>
          <tr><td class="b">Compromised fallback (no group passes 20)</td><td class="warn">25 pts</td><td class="warn">Compromised</td></tr>
          <tr><td class="b">Stretch Match (lone outlier)</td><td class="warn">40 pts</td><td class="warn">Stretch Match</td></tr>
        </tbody>
      </table>
    </div>

    <div class="cl">isValidSplit</div>
    <div class="cb"><pre><span class="kw">function</span> <span class="fn">isValidSplit</span>(
  team1: <span class="st2">Player[]</span>,
  team2: <span class="st2">Player[]</span>,
  maxSpread: <span class="st2">number</span> = <span class="nm">20</span>
): <span class="st2">boolean</span> {
  <span class="kw">const</span> spread = (team: <span class="st2">Player[]</span>) =>
    Math.max(...team.map(p => p.score)) - Math.min(...team.map(p => p.score));
  <span class="kw">return</span> spread(team1) <= maxSpread && spread(team2) <= maxSpread;
}</pre></div>

    <div class="ex">
      <div class="ex-head bad">❌ Example — why this filter exists (real match from today's session)</div>
      <div class="ex-body">
        <p><strong>Sriram (130) + Vivika (78)</strong> vs Yash (98) + Reena (94)</p>
        <p>Team averages: 104 vs 96 — Gap of 8.0 — looks balanced</p>
        <p>Within-team spread Team 1: 130 − 78 = <strong>52 points</strong> — rejected</p>
        <p>Reality: Sriram covers every smash. Vivika cannot contribute. It's 1 vs 2 on key rallies regardless of the average.</p>
      </div>
    </div>

  <div class="section">
    <div class="sl">Part 8 — Ranking</div>
    <div class="st">5-Factor Sort — Applied After All Filters Pass</div>

    <div class="tw">
      <table>
        <thead><tr><th>Priority</th><th>Factor</th><th>Formula</th><th>Direction</th></tr></thead>
        <tbody>
          <tr><td class="b" style="color:var(--blue);">1st</td><td class="b">Tier dispersion</td><td>max(tierIndex of all 4) − min(tierIndex)</td><td>Lower = better. 0 = all same tier.</td></tr>
          <tr><td class="b" style="color:var(--green);">2nd</td><td class="b">Skill gap</td><td>|avg(team1 scores) − avg(team2 scores)|</td><td>Lower = better. This is the primary competitive balance metric. The goal is always the lowest possible gap.</td></tr>
          <tr><td class="b" style="color:var(--amber);">3rd</td><td class="b">Session equity rank</td><td>−sum(sessionDeficit for all 4 players)</td><td>Lower = better. Negated because higher total deficit = more under-served players = should be preferred.</td></tr>
          <tr><td class="b" style="color:var(--red);">4th</td><td class="b">Partner rotation penalty</td><td>+15 per repeated pair this session (cap +30/pair, +60 total)</td><td>Lower = better. Prefer fresh pairings.</td></tr>
          <tr><td class="b" style="color:var(--dim);">5th</td><td class="b">Score variance</td><td>variance of all 4 scores around group mean</td><td>Lower = better. Prefer tighter groups.</td></tr>
        </tbody>
      </table>
    </div>

    <div class="hl"><strong>Tier dispersion is 1st because:</strong> a same-tier game with a gap of 8 is always better than a cross-tier game with a gap of 5. Tier separation is not a tiebreaker — it is the primary filter. Skill gap is 2nd because it directly determines how competitive the game is. Session equity is 3rd — it affects who gets suggested, not whether the game is good. It never overrides a better-balanced game.</div>

    <div class="cl">scoreSplit and rankSuggestions</div>
    <div class="cb"><pre><span class="kw">function</span> <span class="fn">scoreSplit</span>(split: Split, sessionState: SessionState, partnerHistory: PartnerHistory) {
  <span class="kw">const</span> { team1, team2 } = split;
  <span class="kw">const</span> all4 = [...team1, ...team2];

  <span class="cm">// 1. Tier dispersion</span>
  <span class="kw">const</span> tierIndices  = all4.map(p => TIER_INDEX[p.currentTier]);
  <span class="kw">const</span> tierDisp     = Math.max(...tierIndices) - Math.min(...tierIndices);

  <span class="cm">// 2. Skill gap (primary balance metric — always minimise)</span>
  <span class="kw">const</span> avg1         = (team1[0].score + team1[1].score) / 2;
  <span class="kw">const</span> avg2         = (team2[0].score + team2[1].score) / 2;
  <span class="kw">const</span> skillGap     = Math.abs(avg1 - avg2);

  <span class="cm">// 3. Session equity (prefer groups with more under-served players)</span>
  <span class="kw">const</span> sessionAvg   = sessionState.avgGamesPlayed;
  <span class="kw">const</span> totalDeficit = all4.reduce((s, p) => s + (sessionAvg - p.gamesThisSession), 0);
  <span class="kw">const</span> equityRank   = -totalDeficit;  <span class="cm">// negate: higher deficit = better = lower sort value</span>

  <span class="cm">// 4. Partner rotation penalty</span>
  <span class="kw">const</span> pairs = [[team1[0].id, team1[1].id], [team2[0].id, team2[1].id]];
  <span class="kw">let</span> penalty = <span class="nm">0</span>;
  <span class="kw">for</span> (<span class="kw">const</span> pair <span class="kw">of</span> pairs) {
    penalty += Math.min((partnerHistory[<span class="fn">pairKey</span>(pair)] ?? <span class="nm">0</span>) * <span class="nm">15</span>, <span class="nm">30</span>);
  }
  penalty = Math.min(penalty, <span class="nm">60</span>);

  <span class="cm">// 5. Score variance</span>
  <span class="kw">const</span> mean     = all4.reduce((s, p) => s + p.score, <span class="nm">0</span>) / <span class="nm">4</span>;
  <span class="kw">const</span> variance = all4.reduce((s, p) => s + (p.score - mean) ** <span class="nm">2</span>, <span class="nm">0</span>) / <span class="nm">4</span>;

  <span class="kw">return</span> { split, tierDisp, skillGap, equityRank, penalty, variance, avg1, avg2 };
}

<span class="kw">function</span> <span class="fn">rankSuggestions</span>(scored: ScoredSplit[]) {
  <span class="kw">return</span> scored.sort((a, b) =>
    a.tierDisp   - b.tierDisp   ||   <span class="cm">// 1st: same-tier games first</span>
    a.skillGap   - b.skillGap   ||   <span class="cm">// 2nd: smallest team avg gap</span>
    a.equityRank - b.equityRank ||   <span class="cm">// 3rd: most under-served players</span>
    a.penalty    - b.penalty    ||   <span class="cm">// 4th: freshest pairings</span>
    a.variance   - b.variance        <span class="cm">// 5th: tightest group</span>
  ).slice(<span class="nm">0</span>, <span class="nm">5</span>);  <span class="cm">// return top 5 only</span>
}</pre></div>
  </div>
